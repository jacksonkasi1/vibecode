V2: Intelligence Layer Roadmap

The Intelligence Layer

Vibecode is transitioning from a basic, single-agent execution engine to a persistent AI dev team orchestrator. In V2, users talk to a "Central Brain" backed by semantic vector memory, disposable ephemeral workers, and a global Model Context Protocol (MCP) and Skills system. This entirely replaces the heavy, persistent V1 worker polling model.

The Central Brain & Routing

In V1, every prompt spun up a LangGraph worker over Pub/Sub. In V2, the Intelligence Classifier acts as a Central Brain. Before dispatching anything, the Brain intercepts the request. If you ask a question like "Which of my projects uses Stripe?", the Brain resolves it instantly using Vector Search (text-embedding-004) without ever booting an execution VM, saving massive compute and LLM tokens.

Disposable VMs & Cloud Run

In V1, a long-running Node worker endlessly polled Postgres. In the V2 architecture, VMs are completely disposable workers. They spin up, execute a specific task graph in an isolated Git Worktree, sync memory back to Postgres/Vector, and then forcefully self-destruct to prevent disk bloat across 1000s of AI projects.

MCP, Skills & Multi-Agent

V2 replaces simple tools with a unified Capabilities Model. This shared foundation orchestrates Custom Skills (markdown context), Multi-Agent delegation, and Model Context Protocol (MCP) servers at both a Workspace-Level and a User-Global level.

Deep Dive: Intent Classifier

To achieve "instant responses" and save massive amounts of compute, the Hono Server routes every incoming prompt through src/intelligence/classifier.ts before it ever touches a worker VM.

    classDef router fill:#8b5cf6,stroke:#5b21b6,color:white,stroke-width:2px;
    classDef instant fill:#10b981,stroke:#047857,color:white,stroke-width:2px;
    classDef execute fill:#f59e0b,stroke:#b45309,color:white,stroke-width:2px;
    
    Prompt["User Prompt"]:::hnode --> Server{"Hono Server"}:::router
    Server --> Classifier{"Classifier.ts"}:::router
    
    Classifier -->|status_query| P["Postgres DB (Events)"]:::hnode
    P --> UI1["Instant UI Response"]:::instant
    
    Classifier -->|cross_project_search| V["Upstash Vector DB"]:::hnode
    V --> UI2["Instant UI Response"]:::instant
    
    Classifier -->|code_mutation| PS["Pub/Sub Queue"]:::hnode
    PS --> VM["Boot Disposable VM"]:::execute

The Classification Logic:

The classifier analyzes the prompt text and categorizes it into three primary buckets, attaching confidence scores and required context systems: 

export interface ClassificationResult {
  intent: "status_query" | "knowledge_query" | "cross_project_search" | "code_mutation";
  projectId?: string;
  confidence: number;
  requiresExecution: boolean; // If true, spins up a VM
  requiresContext: Array<"postgres" | "vector" | "github_api">;
}
  • status_query: If the user asks "What's the status of my PR?", the classifier flags it, sets requiresExecution: false, pulls recent Postgres events, and answers instantly in the UI. No VM boots.
  • cross_project_search: If the user asks "Which of my projects uses Drizzle ORM?", the classifier flags it, hits the Upstash Vector DB containing the architectural embeddings of all 1000+ user projects, and responds immediately.
  • code_mutation: If the user says "Add dark mode", the classifier sets requiresExecution: true. Only then does the API gateway publish a payload to Pub/Sub to wake up an ephemeral VM.

Deep Dive: Context Hydration

When a VM boots up to process a code_mutation, it wakes up completely amnesiac. Before the LangGraph state machine takes its first step, it runs the Context Hydrator (buildMainSystemPrompt inside deepagents-runtime.ts).

    classDef prompt fill:#8b5cf6,stroke:#5b21b6,color:white;

    Postgres[("💾 Relational DB\n(Chat history, states)")]:::db --> Hydrator
    Vector[("📊 Vector DB\n(Past errors, architecture)")]:::db --> Hydrator
    Config["📁 Capabilities\n(AGENTS.md, skills/)"] --> Hydrator
    
    Hydrator["💧 Context Hydrator"] --> SystemPrompt["📜 Final System Prompt"]:::prompt
    SystemPrompt --> LangGraph

Dynamic Injection:

Instead of dumping the entire repository into the prompt, the Hydrator surgically injects only what matters:

  • Editor State: The frontend continuously syncs the user's IDE state. The Hydrator specifically injects: "Hidden context: The user is currently looking at lines 40-80 of src/components/Button.tsx." This allows the AI to understand phrases like "fix this file".
  • File Discovery: It runs detectSkillSources() and detectMemorySources() in the worktree, finding global .agents/ folders and dynamically appending them to the system ruleset.
  • Vector Recall: It queries Upstash using the prompt to find "How we fixed a similar bug in this project 3 weeks ago" and injects the solution so the AI doesn't repeat past mistakes.

Deep Dive: LangGraph Orchestrator

Vibecode V2 abandons the standard "ReAct" single-agent loop. Once the Context Hydrator prepares the prompt, it hands control to a sophisticated LangGraph State Machine designed for Multi-Agent delegation and mandatory code validation.

The Graph Flow:

Key Orchestration Mechanics:

  • Sub-Agent Bounding: The Orchestrator does not write code. It acts as a tech lead. If the plan requires a database change and a UI change, it delegates to the Database Sub-Agent (injecting only Drizzle ORM skills) and the Frontend Sub-Agent (injecting only React/Tailwind skills). This keeps LLM context windows small and highly focused, preventing hallucination.
  • Sandbox Interfacing (LocalShellBackend): When a Sub-Agent needs to write a file or run a test, it hits the Tool Node. This node strictly executes the command inside the Git Worktree sandbox (with a 120s timeout and 100KB output limit), piping the real terminal output back into the sub-agent's state array.
  • The Validation Gate Loop: The graph cannot reach the End state until it passes the Validation Gate (runValidationGate). The gate parses the user's package.json, looks for check-types or build scripts, and executes them. If they fail, the graph routes to an Error Fixer Node, appending the exact compiler errors to the state, forcing the AI to fix its own typos before returning to the user.

Deep Dive: Git Worktree Sandboxing

When a disposable VM boots up, it doesn't just clone the user's repository normally. Doing so for 1,000s of projects would be slow and bloated. Instead, the Worker relies on Git Worktrees to instantly attach a sandboxed folder (/tmp/workspace_xyz) directly to the central bare Git data.

Granular Checkpointing & Orphaned Commits:

Vibecode V2 decouples its "AI Memory" from the user's actual Git branch. It doesn't clutter the main branch with terrible "AI attempted fix" commits. 

gitGraph
    commit id: "Initial Commit"
    commit id: "User: setup layout"
    commit id: "User: add button"
    branch refs/checkpoints/exec_123
    checkout refs/checkpoints/exec_123
    commit id: "AI: plan UI"
    commit id: "AI: install npm package"

    checkout main
    commit id: "User: PR merge"
  • Custom Refs: Every single step the AI takes generates an orphaned commit (using low-level git write-tree). These are stored invisibly under refs/checkpoints/<execution_id>.
  • Perfect Snapshots: This captures the exact state of the index and the physical files. The commit SHAs are logged to the workspace_revision Postgres table.

Two-Phase Rollbacks & Restoration:

If an AI Agent hallucinates badly or deletes a critical file, the system must restore state perfectly. It does not use git reset --hard (which is slow and prone to dangling file bugs). It executes a strict Two-Phase Rollback via plumbing commands: 

# 1. Update the index to match the target rollback tree
git read-tree <worktreeTree_Hash>

# 2. Force checkout the index into the physical filesystem
git checkout-index -a -f

# 3. Clean up any untracked garbage files left behind
git clean -fd

Phase 1: Attempt the physical Git state restoration first.
Phase 2: Only if Git succeeds, update the database (marking executions as reverted). If Git fails, abort the DB transaction to prevent UI desync.

Aggressive Physical Cleanup:

When the Disposable VM finishes the job, it doesn't just shut down. It actively protects the cloud disk by running:

  • Worktree Teardown: git worktree remove <path> --force instantly deletes the sandbox files.
  • Process Murder: Runs a force SIGKILL on any lingering terminal commands started by the AI.
  • Cache Purging: Wipes leftover .log artifacts to prevent silent storage leaks over time.
