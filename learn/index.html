Current Architecture

This document outlines the current implementation of Vibecode. It details how the web client, API server, and long-running worker orchestrate code generation using strict Git Worktree isolation and Pub/Sub queuing.

💻

apps/web

Stack: React 19, Vite, Tailwind v4, Zustand. Uses `@assistant-ui/react` and `@streamdown` for rich, live-streaming markdown and IDE-like code blocks. React Query handles API state.

⚙️

apps/server

Stack: Hono, Bun. Acts as the API gateway. It manages REST endpoints, handles Better-Auth, and importantly, tails the Postgres DB using a cursor to push Server-Sent Events (SSE) live to the frontend so users can watch the AI agent think.

🧠

apps/worker

Stack: LangChain, LangGraph, DeepAgents. The core engine running autonomously. It pulls jobs from Pub/Sub, orchestrates tasks, and executes real terminal commands using Node Child Processes.

packages/db

Stack: Drizzle ORM, Neon Serverless Postgres.

The Persistent Relational Memory Layer ensures context is never lost and work is tracked safely. This is the source of truth that differentiates Vibecode from a standard memory-less chat window.

  • Conversational Context: chat_thread & chat_message maintain history across sessions so the AI recalls past architectural decisions.
  • Sandbox Revisions: workspace_revision tracks isolated sandbox versions. Every execution records a commitHash so the agent can technically roll back or branch off past iterations.
  • Real-Time State Tracking: executions represent LangGraph tasks. As the agent moves between sub-agents and runs shell tools, it streams JSON logs into execution_events for the server to tail via SSE.

Pub/Sub Message Flow

When the Server receives a prompt, it writes it to Postgres and uses @google-cloud/pubsub to publish a tiny JSON payload containing only the IDs. The actual prompt text is not sent in the payload, ensuring it stays well within queue message limits. 

{
  "executionId": "exec_abc123",
  "workspaceId": "ws_xyz789",
  "userId": "usr_dev01",
  "queuedAt": "2026-03-26T00:01:21.000Z"
}

The worker constantly listens to the subscription. Once it claims the executionId from the DB, it calls message.ack() to remove it from the queue. Flow control is set to maxMessages: 4, meaning a single worker VM will only ever run 4 LangGraph jobs simultaneously to avoid memory crashes.

🛡️

Sandbox Rules & Linux

Running AI-generated shell commands on a VM is dangerous. Vibecode enforces strict rules using the deepagents plugin's LocalShellBackend:

  • Strict Directory Jailing: The AI's cwd is completely locked to the Git Worktree (/tmp/vibecode-xyz/).
  • Process Timeouts: If the AI runs a command that hangs indefinitely (like starting a web server or infinite loop), the LocalShellBackend kills the process after 120 seconds and returns a timeout error to the AI.
  • Output Flooding Limits: stdout/stderr buffers are truncated at exactly 100,000 bytes to stop memory flooding.
🔄

Local Development (Poller)

Google Cloud Pub/Sub is fantastic for production, but terrible for local development. Vibecode includes a seamless Poller Fallback. If WORKER_DISPATCH_MODE is not "pubsub", the worker gracefully degrades into an infinite setInterval loop that queries Postgres for stuck jobs every few seconds. This makes the repo entirely plug-and-play for local contributors.

Git Worktree Architecture

To prevent AI agents from breaking the main codebase while they "think" and test code, Vibecode relies heavily on Git Worktrees. A worktree allows multiple working directories to be attached to a single Git repository. They share the same .git folder, meaning the AI gets an instantly cloned folder on the disk with zero overhead.

    Repo -->|git worktree add| WT1["/tmp/workspace_xyz\n(Branch: ai/execution-123)"]:::worktree
    Repo -->|git worktree add| WT2["/tmp/workspace_abc\n(Branch: ai/execution-456)"]:::worktree

Execution Isolation Flow:

  1. Setup: Before the AI starts, the worker creates a physical isolation zone: git worktree add /tmp/workspace_xyz -b ai/execution-123.
  2. Jailed Execution: The LangGraph agent runs shell commands (npm test, tsc) strictly inside /tmp/workspace_xyz. Your open IDE window is entirely unaffected by hallucinated syntax errors or infinite loops.
  3. Checkpointing: Once successful, the agent commits the work. The resulting commitHash is securely saved into the Postgres workspace_revision table alongside the execution ID.
  4. Cleanup: The worktree is forcibly removed (git worktree remove --force) when the job is done to reclaim disk space instantly.

Two-Phase Safe Rollbacks:

If the validation gate fails, or the user clicks "Undo" in the UI, Vibecode doesn't use a messy git reset --hard. Instead, it uses strict, low-level Git plumbing commands for a faster and safer Two-Phase Rollback: 

git read-tree <parentHash>
git checkout-index -a -f
git clean -fd

Phase 1: Attempt the physical Git state restoration first.
Phase 2: Only if Git succeeds, update the database (marking executions as reverted). If Git fails, the DB transaction aborts to prevent the UI from ever falling out of sync with the physical disk.
